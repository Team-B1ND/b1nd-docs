# Team-B1ND GitHub 이슈 관리 가이드

이 스킬은 Team-B1ND의 이슈 관리 규칙과 라벨 시스템을 자동으로 적용하여 체계적인 작업 추적과 관리를 도와줍니다.

## 이슈 제목 규칙

### 기본 형식

```
<description>
```

### 작성 규칙

1. **한국어, 영어 모두 사용 가능**
2. **명확하고 구체적으로** 작성
3. **50자 이내**로 간결하게
4. **마침표 사용 금지**
5. **라벨로 타입을 표시하더라도 제목에 type 포함 권장**

### 좋은 예시

```
웹뷰 브릿지 카메라 API 구현
로그인 시 토큰 갱신 오류 수정
알림 서비스 MSA로 분리
API 명세서 업데이트
Docker 환경 설정 추가
검색 속도 개선
E2E 테스트 추가
프로덕션 서버 다운 긴급 수정
```

### 나쁜 예시

```
❌ 기능 추가 → 너무 모호함
❌ 카메라 기능을 추가하고 싶어요. → 마침표, 구어체
❌ 웹뷰 브릿지 카메라 API를 구현하고 네이티브 통신을 추가하며... → 너무 김
```

## 필수 라벨 시스템

모든 이슈는 **3가지 필수 라벨**을 가져야 합니다:
1. **Type 라벨** - 이슈 유형
2. **Priority 라벨** - 우선순위
3. **Size 라벨** - 작업량

### 1. Type 라벨 (필수)

| 라벨 | 설명 | 예시 |
|------|------|------|
| `type: feature` | 새로운 기능 추가 요청 | 웹뷰 브릿지 카메라 API 구현 |
| `type: bug` | 의도하지 않은 동작/오류 | 로그인 시 토큰 갱신 오류 수정 |
| `type: refactor` | 기능 변경 없이 코드 구조 개선 | 알림 서비스 MSA로 분리 |
| `type: docs` | 문서 추가, 수정, 개선 | API 명세서 업데이트 |
| `type: chore` | 빌드, 설정, 패키지 등 유지보수 | Docker 환경 설정 추가 |
| `type: enhancement` | 기존 기능의 개선 및 향상 | 검색 속도 개선 |
| `type: test` | 테스트 코드 추가 또는 수정 | E2E 테스트 추가 |
| `type: hotfix` | 프로덕션 환경의 긴급 버그 수정 | 프로덕션 서버 다운 긴급 수정 |

### 2. Priority 라벨 (필수)

| 라벨 | 설명 | 예시 시나리오 |
|------|------|--------------|
| `priority: critical` | **긴급** - 서비스 중단 또는 심각한 버그로 즉시 처리 필요 | 프로덕션 서버 다운, 데이터 손실 |
| `priority: high` | **높음** - 중요한 기능 또는 주요 버그로 우선 처리 | 핵심 기능 버그, 중요 기능 추가 |
| `priority: medium` | **보통** - 일반적인 작업으로 정상 일정으로 처리 | 일반 기능 개선, 마이너 버그 |
| `priority: low` | **낮음** - 급하지 않은 작업으로 여유있을 때 처리 | 문서 개선, UI 세부 조정 |

### 3. Size 라벨 (필수)

| 라벨 | 예상 작업 시간 | 설명 |
|------|--------------|------|
| `size: XS` | **4시간 이내** | 매우 작음 - 간단한 수정 |
| `size: S` | **1-2일** | 작음 - 작은 기능이나 버그 수정 |
| `size: M` | **3-5일** | 보통 - 일반적인 기능 개발 |
| `size: L` | **1-2주** | 큼 - 복잡한 기능이나 대규모 리팩토링 |
| `size: XL` | **2주 이상** | 매우 큼 - 주요 기능 (분할 검토 필요) |

**중요**: XL 크기 이슈는 여러 개의 작은 이슈로 분할을 검토해야 합니다.

### 4. Init 라벨 (선택)

초기 설정 작업에만 사용:

| 라벨 | 설명 | 예시 |
|------|------|------|
| `init: project` | 프로젝트 초기화 - 새 프로젝트/모듈 초기 설정 | 프로젝트 스캐폴딩 |
| `init: structure` | 구조 초기화 - 폴더 구조, 아키텍처 설정 | 디렉토리 구조 설정 |
| `init: docs` | 문서 초기화 - README, 기여 가이드 등 기본 문서 | README 작성 |

## 이슈 본문 템플릿

### 기본 구조

```markdown
## 작업 내용
(무엇을 할 것인지 명확하게 설명합니다)

## 작업 목적
(왜 이 작업이 필요한지 설명합니다)

## 체크리스트
- [ ] 작업 항목 1
- [ ] 작업 항목 2
- [ ] 작업 항목 3

## 참고 자료
(관련 링크나 자료가 있다면 첨부합니다)
```

### 작성 규칙

1. **~니다 체 사용** (~합니다, ~입니다)
2. **템플릿 주석 지우지 말 것**
3. **템플릿의 모든 칸 작성**
4. **명확하고 구체적으로** 작성

### 좋은 예시

```markdown
제목: 웹뷰 브릿지 카메라 API 구현
라벨: type: feature, priority: high, size: M

## 작업 내용
모바일 앱에서 웹뷰와 네이티브 카메라를 연동하는 브릿지 API를 구현합니다.

## 작업 목적
사용자가 웹뷰 내에서 카메라 기능을 사용할 수 있도록 합니다.

## 체크리스트
- [ ] 카메라 권한 요청 로직 구현
- [ ] 브릿지 API 인터페이스 설계
- [ ] 웹뷰-네이티브 통신 구현
- [ ] 이미지 전송 로직 구현
- [ ] 테스트 코드 작성

## 참고 자료
- [웹뷰 브릿지 가이드](링크)
```

## 이슈 할당

- 이슈 생성 시 담당자(Assignee) 지정
- 담당자는 **매주 트리아지 회의**를 통해 배정
- 초기 생성 시 미할당 가능, 트리아지 회의에서 할당

## 스킬 적용 시나리오

### 1. 이슈 생성 요청 시

사용자가 "이슈 만들어줘", "GitHub 이슈 생성" 등을 요청할 때:
1. 이슈 제목 작성 (50자 이내, 명확하고 구체적)
2. Type 라벨 선택 (feature, bug, refactor 등)
3. Priority 라벨 선택 (critical, high, medium, low)
4. Size 라벨 선택 (XS, S, M, L, XL)
5. 이슈 본문 작성 (~니다 체)
6. `gh issue create` 실행

### 2. 라벨 자동 선택 가이드

**Type 선택**:
```
"새 기능" → type: feature
"버그", "오류" → type: bug
"리팩토링" → type: refactor
"문서" → type: docs
"설정", "빌드" → type: chore
"개선" → type: enhancement
"테스트" → type: test
"긴급", "프로덕션" → type: hotfix
```

**Priority 자동 판단**:
```
"서비스 중단", "프로덕션 다운" → priority: critical
"중요", "핵심 기능" → priority: high
"일반", "개선" → priority: medium
"나중에", "여유" → priority: low
```

**Size 추정**:
```
"간단한", "빠른 수정" → size: XS
"작은 기능" → size: S
"일반 기능" → size: M
"복잡한", "대규모" → size: L
"주요 기능", "여러 개로 나눠야" → size: XL
```

### 3. 이슈 검증

이슈 생성 시 자동으로 다음을 검증:
- ✅ 제목이 50자 이내인가?
- ✅ 제목이 명확하고 구체적인가?
- ✅ 마침표가 없는가?
- ✅ Type 라벨이 지정되었는가?
- ✅ Priority 라벨이 지정되었는가?
- ✅ Size 라벨이 지정되었는가?
- ✅ 본문의 모든 섹션이 작성되었는가?

### 4. XL 이슈 분할 제안

Size가 XL로 추정되면:
1. 경고 메시지 표시
2. 더 작은 이슈로 분할 제안
3. 분할 가능한 단위 제시

## 라벨 조합 예시

| 이슈 유형 | Type | Priority | Size | 예시 |
|----------|------|----------|------|------|
| 긴급 버그 수정 | bug | critical | S | 프로덕션 로그인 오류 |
| 신규 기능 | feature | high | M | 웹뷰 카메라 API |
| 문서 개선 | docs | low | XS | README 오타 수정 |
| 대규모 리팩토링 | refactor | medium | L | MSA 분리 |
| 테스트 추가 | test | medium | S | E2E 테스트 |

## 금지사항

- ❌ 의미없는 제목 (ex: 테스트, 버그 수정)
- ❌ 본문 없이 제목만 작성
- ❌ **라벨 미지정** (3가지 필수 라벨 필수)
- ❌ 하나의 이슈에 여러 관련없는 작업 포함

## 자동 적용 프로세스

1. 이슈 생성 요청 감지
2. 제목 작성 (50자 이내, 명확)
3. 작업 내용 분석하여 Type 라벨 자동 선택
4. 긴급도 분석하여 Priority 라벨 자동 선택
5. 작업량 추정하여 Size 라벨 자동 선택
6. 이슈 본문 템플릿 자동 작성 (~니다 체)
7. gh CLI로 이슈 생성 (라벨 포함)
8. 이슈 번호와 URL 반환

## 이슈와 브랜치/PR 연결

- 각 이슈는 **하나의 브랜치**와 1:1 매핑
- 브랜치명에 이슈 번호 포함: `feature/#123-camera-api`
- PR 제목에 이슈 번호 포함: `[Feature] 카메라 API (#123)`
- PR 본문에 `Resolves #123` 추가
