Team-B1ND PR 템플릿에 맞춰 Pull Request를 생성해주세요.

## PR 제목 규칙

**형식**: `[브랜치명] (#이슈번호)`

**작성 규칙**:
1. 한국어, 영어 모두 사용 가능
2. **이슈 번호를 반드시 포함** (예: #123)
3. **50자 이내**로 작성
4. 마침표 사용 금지

**좋은 예시**:
```
[Feature] 웹뷰 브릿지 카메라 API 구현 (#123)
[Fix] 로그인 시 토큰 갱신 오류 수정 (#456)
[Refactor] 알림 서비스 MSA로 분리 (#789)
```

**나쁜 예시**:
```
기능 추가 → 브랜치명 미표기, 이슈번호 누락
[Feature] 카메라 기능을 추가했습니다. (#123) → 마침표, 과거형
[Feature] 웹뷰 브릿지 카메라 API를 구현하고... (#123) → 너무 김
```

## PR 본문 구조

### 필수 항목 ✅

```markdown
## 변경 사항
무엇을 변경했는지 명확하게 설명합니다.

## 작업 내용
어떻게 구현했는지 설명합니다.

## 테스트 방법
리뷰어가 어떻게 테스트할 수 있는지 설명합니다.
- [ ] 단위 테스트 추가/수정
- [ ] 통합 테스트 완료
- [ ] 수동 테스트 완료

## 관련 이슈
Resolves #이슈번호
```

### 선택 항목 (필요시 추가)

```markdown
## 스크린샷/영상 (UI 변경 시)
Before/After를 첨부합니다.

## 성능 변화 (성능 개선 작업 시)
측정 결과를 첨부합니다.

## Breaking Changes
하위 호환성이 깨지는 변경이 있다면 명시합니다.

## 참고 사항
리뷰어가 알아야 할 추가 정보를 작성합니다.
```

### 체크리스트

```markdown
## 체크리스트
- [ ] 코드가 프로젝트의 코딩 스타일을 따릅니다
- [ ] 자체 코드 리뷰를 완료했습니다
- [ ] 문서를 업데이트했습니다 (필요한 경우)
- [ ] Breaking Changes가 없습니다
```

## 작업 순서

1. `git status`로 현재 브랜치 확인
2. `git log` 및 `git diff [타겟브랜치]...HEAD`로 모든 커밋 분석
3. 변경 파일이 10개 이상이면 PR 분리 검토
4. PR 제목 작성: `[브랜치명] (#이슈번호)` 형식 (50자 이내)
5. **필수 항목** 모두 작성: 변경 사항, 작업 내용, 테스트 방법, 관련 이슈
6. 선택 항목 필요 시 추가
7. `gh pr create` 명령으로 PR 생성 (HEREDOC 사용 필수)

## 인자

타겟 브랜치: $ARGUMENTS (기본값: main)

## PR 크기 가이드라인

- **하나의 PR = 하나의 이슈**
- 변경 파일 **10개 이상**이면 분리 검토
- 리뷰어가 **30분 이내** 리뷰 가능한 크기

## 리뷰 요청

- 최소 1명 이상 리뷰어 지정
- 도메인 전문가/코드 오너 포함
- 리뷰 완료 전까지 머지 금지

## 금지사항

- ❌ 의미없는 제목 (ex: Feature 작업, Bug 수정)
- ❌ 본문 없이 제목만 작성
- ❌ 테스트하지 않은 코드 PR
- ❌ 리뷰어 피드백 없이 강제 머지

PR 생성 후 URL을 알려주세요.
